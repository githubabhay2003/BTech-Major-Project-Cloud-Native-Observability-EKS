Project Scope

What This Platform Does

The system deploys a FastAPI backend and an NGINX-served static website on Amazon EKS, then wraps them in a complete observability stack — all provisioned through code and delivered through a fully automated Continuous Integration and Continuous Deployment pipeline triggered on every push to the main branch.

01 — Metrics Collection

Prometheus

The FastAPI backend is instrumented with prometheus-fastapi-instrumentator, automatically exposing a /metrics endpoint. A Kubernetes ServiceMonitor scrapes it every 15 seconds alongside cluster-wide node and pod metrics. Retention period: 7 days.

02 — Visualization

Grafana

Custom Golden Signals dashboards defined as JSON, stored in a Kubernetes ConfigMap, and auto-loaded via a Grafana sidecar. Surfaces request rate, error rate, latency percentiles (P95 and P99), CPU usage, and memory usage in real time.

03 — Alerting

Alertmanager

PrometheusRule custom resources define alert conditions for high error rate, high latency, and service-down scenarios. Alertmanager groups, deduplicates, and delivers notifications via Simple Mail Transfer Protocol with Warning and Critical severity levels.

04 — External Monitoring

Blackbox Exporter

A Kubernetes Probe resource configures the Prometheus Blackbox Exporter to send periodic HTTP requests to the website from outside the application layer, simulating real user access and triggering alerts if the service becomes unreachable.

Application Layer

FastAPI backend (Python 3.11-slim, Uvicorn on port 8000) with 2 replicas, liveness and readiness probes on /health, ClusterIP service port 80 → 8000. Static website via NGINX Alpine on port 80. Both images built via Dockerfiles and pushed to Amazon Elastic Container Registry tagged by Git commit SHA.

Observability Layer

Deployed in the monitoring namespace via kube-prometheus-stack Helm chart. Prometheus retains 7 days of time-series data. Grafana loads dashboards from a ConfigMap sidecar. Blackbox Exporter checks external website availability. Monitoring tools exposed under /grafana, /prometheus, and /alertmanager.

Infrastructure Layer

Two Terraform execution layers: terraform/infra provisions Virtual Private Cloud, subnets, gateways, Amazon EKS cluster, Identity and Access Management roles, Amazon Elastic Container Registry, and Bastion Host. terraform/apps reads remote state outputs and deploys NGINX Ingress Controller and full observability stack.

Continuous Integration and Continuous Deployment

End-to-End Deployment Pipeline

A push to the main branch triggers the entire pipeline. No manual deployment steps exist after initial infrastructure provisioning.

01

Provision Infrastructure with Terraform

The terraform/infra layer provisions the Amazon Virtual Private Cloud (10.0.0.0/16, two public and two private subnets across two Availability Zones), Internet Gateway, NAT Gateway, Amazon EKS cluster named eks-observability-cluster, Identity and Access Management roles for cluster, node group, GitHub Actions (OpenID Connect), and Bastion Host. Remote state is written to Amazon S3 with DynamoDB locking.

terraform init → terraform plan → terraform apply
02

Configure Kubernetes Access

AWS Command Line Interface updates the local kubeconfig to point at the newly created cluster. The aws-auth ConfigMap is updated to grant Identity and Access Management access to worker nodes and the GitHub Actions role. Access is verified with kubectl get nodes.

aws eks update-kubeconfig --name eks-observability-cluster
03

Deploy Platform Components via Terraform Applications Layer

The terraform/apps layer reads remote state outputs and deploys NGINX Ingress Controller (LoadBalancer type), the full kube-prometheus-stack (Prometheus with 7-day retention, Grafana with default dashboards, Alertmanager), Blackbox Exporter, and ingress resources routing monitoring tools under /grafana, /prometheus, and /alertmanager.

terraform apply (apps layer)
04

Authenticate to Amazon Web Services via OpenID Connect

GitHub Actions uses the id-token: write permission to request a JSON Web Token from GitHub's OpenID Connect provider. The token is exchanged for temporary AWS credentials by assuming the eks-observability-github-actions-role. No static access keys are stored in the repository at any point.

aws-actions/configure-aws-credentials@v4 · OpenID Connect
05

Build and Push Docker Images to Amazon Elastic Container Registry

Two images are built: ./apps/fastapi (Python 3.11-slim, Uvicorn on port 8000) and ./apps/website (NGINX Alpine on port 80). Each is tagged with the commit SHA and pushed to their respective repositories in account in the us-east-1 region.

docker build · docker tag · docker push (Amazon ECR)
06

Deploy Applications via Helm Charts

Two Helm charts are applied: helm/fastapi (2 replicas, ClusterIP service on port 80 → 8000, liveness and readiness probes on /health, ingress rewrite on /api) and helm/website (ingress on /). Dynamic values pass the Amazon Elastic Container Registry repository URL and commit SHA image tag for full version traceability.

helm upgrade --install fastapi-app ./helm/fastapi --set image.tag=$IMAGE_TAG

Monitoring and Alerting

Observability Implementation

All monitoring tools are deployed in the monitoring namespace and exposed through path-based routing on the NGINX Ingress Controller LoadBalancer endpoint.

Prometheus

Collects metrics from FastAPI /metrics, Kubernetes nodes (Node Exporter), pods (kube-state-metrics), and all cluster components. Retention period: 7 days. Scrape interval: 15 seconds. One replica deployed in the monitoring namespace. Evaluates PrometheusRule alert conditions continuously.

Running in monitoring namespace · port 9090
Ingress path: /prometheus
Grafana

Displays the Golden Signals dashboard — Traffic (Requests Per Second), Error Rate (5xx percentage), Latency (P95 and P99 percentiles), CPU usage, and memory consumption. Dashboard definitions stored as JSON in a Kubernetes ConfigMap and auto-loaded by a Grafana sidecar container on startup.

Default dashboards enabled · port 3000
Ingress path: /grafana
Alertmanager

Receives firing alerts from Prometheus, groups and deduplicates them, then routes notifications via Simple Mail Transfer Protocol email. Severity levels: Warning (performance degradation) and Critical (service unavailability). Includes documented handling for expected Amazon EKS managed-component alerts.

Simple Mail Transfer Protocol routing configured
Ingress path: /alertmanager
Blackbox Exporter

A Kubernetes Probe resource directs the Prometheus Blackbox Exporter to send HTTP GET requests to the website endpoint from outside the application layer, measuring response success and availability. Triggers an alert via Alertmanager when the probe fails, simulating real end-user access.

Probe resource active · HTTP module
Target: website service via Ingress

Engineering Insights

Key Challenges and Resolutions

Fifteen critical challenges were resolved across infrastructure, Kubernetes, Continuous Integration and Deployment, and observability layers. A complete document covering over 100 cases is available in the project repository at docs/challenges-and-learnings.md.

Challenge 01
Cross-Module Resource Referencing Failure
Terraform execution failed when modules tried to access each other directly. Resolved by using outputs from the infrastructure layer and passing them as explicit inputs to the applications layer.
Terraform · Modules
Challenge 02
Amazon EKS Access Failure — Role-Based Access Control
Cluster access failed for users and the Continuous Integration pipeline because Identity and Access Management roles were not mapped. Resolved by configuring the aws-auth ConfigMap with correct Role-Based Access Control mappings.
Amazon EKS · Authentication
Challenge 03
Remote State Dependency Issues
The applications Terraform layer could not fetch outputs from the infrastructure layer due to incorrect terraform_remote_state backend configuration. Fixed by aligning backend configuration and explicitly declaring all required outputs.
Terraform · State Management
Challenge 04
Custom Resource Definitions Not Installed Before Usage
Observability resources (ServiceMonitor, PrometheusRule) failed to apply because their Custom Resource Definitions were not yet installed. Resolved by enforcing correct dependency ordering using depends_on in Terraform.
Kubernetes · Custom Resource Definitions
Challenge 05
FastAPI Metrics Endpoint Not Available
Prometheus could not scrape application metrics because no /metrics endpoint existed. Resolved by adding prometheus-fastapi-instrumentator and calling Instrumentator().instrument(app).expose(app) in main.py.
FastAPI · Observability
Challenge 06
Label Mismatch in Kubernetes
Kubernetes label selectors are case-sensitive. Mismatches such as app: FastAPI versus app: fastapi broke the entire ServiceMonitor target binding. All labels were standardised across Helm chart templates and values files.
Kubernetes · Labels
Challenge 07
Image Pull Failures — ImagePullBackOff
Pods failed to start because the Amazon Elastic Container Registry image URI was incorrect or the worker node role lacked AmazonEC2ContainerRegistryReadOnly permission. Both the URI format and the Identity and Access Management role policy attachment were corrected.
Kubernetes · Amazon ECR
Challenge 08
Continuous Integration Authentication Complexity
Difficulty securing AWS access from GitHub Actions without exposing static credentials. Resolved by implementing OpenID Connect-based authentication via aws-actions/configure-aws-credentials@v4, eliminating all hardcoded AWS access keys.
Identity and Access Management · OpenID Connect
Challenge 09
Helm Release Stuck in Failed State
A previous failed Helm deployment left stale metadata that blocked all subsequent deployments for the same release name. The failed release was cleaned up manually and re-deployment succeeded with corrected chart values.
Helm
Challenge 10
Terraform Destroy Failures Due to AWS Dependencies
AWS resources including Elastic Load Balancers and Elastic Network Interfaces remained attached after Kubernetes resources were deleted, causing terraform destroy to fail. Kubernetes Ingress and Service resources must be deleted before running destroy.
Amazon Web Services · Terraform
